# AI Quant Lab 平台升级 Spec

> 状态：Draft v1.0
> 日期：2026-07-11
> 范围：产品、体验与技术架构规划；本阶段不实现
> 面向版本：AI Quant Lab 2.0

## 1. 项目愿景

将 `ai-quant-lab` 从单一的 Indicator Studio 升级为一个可长期扩展的个人量化策略网站：它既是专业、优美的策略展示主页，也是策略研究、回测、参数分析和实验沉淀的统一入口。

平台应做到：

- 新策略以标准化模块接入，而不是继续在单个页面中堆叠条件分支。
- 每个策略拥有独立、可分享、可复现的详情页。
- 首页能快速表达平台定位、策略覆盖范围、研究纪律和近期成果。
- 公开访问时不依赖本机 RQData 授权；本地研究时可使用完整数据与回测能力。
- 已有双均线、海龟策略、指标研究、参数优化和 `quant_lab` 回测核心能够迁移复用。
- 视觉风格保持克制、可信、专业，不做交易所终端式的信息堆砌。

## 2. 当前状态与主要问题

### 2.1 可复用资产

- `indicator-studio/` 已具备行情、指标、信号、对比、优化、回测、模板、AI 解释和项目实验功能。
- `quant_lab/` 已分离数据、策略、回测、指标、优化、绘图和 RQAlpha 适配层。
- 已有策略：`ma_cross`、`turtle_breakout`。
- 本地 Python 服务已有健康检查、行情、策略目录、回测、优化和解释接口。
- 已明确“收盘确认、下一交易日开盘成交”等避免未来函数的基础约束。

### 2.2 需要解决的问题

- 当前没有平台首页，用户进入后直接面对复杂研究工具。
- 所有前端功能集中在一个 HTML、一个大型 JavaScript 文件和一个 CSS 文件中，新增策略成本会持续上升。
- 策略参数在前端、后端、轻量回测、RQAlpha 和草稿生成等位置重复硬编码。
- “指标工具”“策略详情”“回测实验”“研究项目”混在同一个标签页层级，信息架构不清晰。
- localStorage 适合临时配置，不适合长期策略版本、实验结果和研究笔记管理。
- 公开网站与本地专业服务的边界不明确，部署后真实数据和授权能力不可控。
- 当前页面对移动端、无障碍、错误状态和空状态的设计不够系统。

## 3. 产品定位

### 3.1 一句话定位

AI Quant Lab 是一个以可复现研究为核心的个人量化策略库与验证平台。

### 3.2 目标用户

首要用户是平台所有者本人，同时兼顾三类访问者：

1. 研究者：查看策略逻辑、调整参数、执行回测并比较实验。
2. 学习者：理解策略为何产生信号、适用什么行情、有什么风险。
3. 访客：通过公开页面快速了解策略库、研究方法和代表性结果。

### 3.3 产品原则

- 先解释策略，再展示收益。
- 任何业绩都同时展示区间、基准、成本假设和最大回撤。
- 明确区分历史回测、样本外验证、模拟运行和实盘表现。
- 页面默认展示稳定性与风险，不默认强调“最优参数”。
- 每次回测都能通过配置快照复现。
- 新策略应通过声明式元数据和标准接口接入。

## 4. 范围

### 4.1 2.0 首期范围

- 平台首页。
- 策略目录与筛选。
- 双均线、海龟策略独立详情页。
- 通用策略研究工作台。
- Indicator Studio 迁移为独立工具页。
- 统一策略注册表和参数 Schema。
- 统一回测请求、结果和错误协议。
- 本地研究模式与公开展示模式。
- 实验的创建、保存、查看和对比。
- 响应式布局、深浅主题、基础无障碍和风险声明。

### 4.2 暂不进入首期

- 自动实盘交易或券商下单。
- 多用户社区、评论、跟单和策略售卖。
- 承诺收益、智能荐股或个性化投资建议。
- 分布式回测集群和大规模因子计算平台。
- 在浏览器中保存 RQData、Tushare 等凭据。
- 在没有真实记录时展示伪造的实时收益或策略排名。

## 6. 首页 Spec

### 6.1 首页目标

访问者在首屏内应理解三件事：这是量化策略研究平台；策略可以被验证和复现；平台已有可进入的策略内容。

### 6.2 页面结构

1. 顶部导航
   - 品牌标识 `AQ / AI Quant Lab`。
   - 主导航、主题切换、本地服务状态。
   - 主按钮“浏览策略”，次按钮“进入研究室”。

2. Hero 首屏
   - 主标题建议：`把每一个策略，变成可解释、可验证、可复现的研究。`
   - 一段简洁副文案，明确不提供收益承诺。
   - 右侧使用抽象的策略表现面板：净值曲线、回撤带、实验标签；使用示例数据时必须标注。
   - 不使用夸张金融大图、金币、火箭或霓虹交易屏素材。

3. 平台概览
   - 已收录策略数、已保存实验数、覆盖资产类别、最近研究日期。
   - 数据不足时显示真实的零状态，不填充虚假数字。

4. 精选策略
   - 首期展示双均线与海龟策略。
   - 卡片包含类别、状态、周期、核心逻辑、风险级别、最近验证时间和进入按钮。
   - 收益数据仅在存在固定验证快照时出现，并附区间与基准。

5. 研究流程
   - 用四步表达：提出假设 → 定义规则 → 历史验证 → 稳健性检查。

6. 最新实验
   - 展示最近本地实验；公开模式展示经过发布的只读快照。
   - 每条包含策略、标的、区间、引擎、收益、回撤和创建时间。

7. 方法与风险
   - 突出交易成本、未来函数、幸存者偏差、样本外验证和参数过拟合。

8. 页脚
   - 版本、数据来源说明、免责声明、Git/文档入口（若未来公开）。

## 7. 策略库与策略详情

### 7.1 策略库

支持按以下维度筛选：

- 类型：趋势、突破、均值回归、动量、多因子、资产配置。
- 市场：A 股、美股、ETF、期货、加密资产；未实现的类别不显示。
- 周期：日频、周频、分钟级。
- 状态：草稿、研究中、已验证、已归档。
- 风险：低、中、高。

排序支持：最近更新、名称、策略状态。首期不提供按收益率排序，避免鼓励无上下文的业绩比较。

### 7.2 策略卡片统一字段

- 名称、英文标识和一句话摘要。
- 策略类别和适用市场。
- 研究状态与版本。
- 主要持有周期。
- 关键风险标签。
- 最近验证时间。
- “查看策略”和“开始研究”入口。

### 7.3 策略详情页

每个策略详情页使用同一页面骨架，策略模块只提供内容与配置：

1. 策略概览：假设、版本、状态、适用市场、周期和风险。
2. 规则说明：入场、退出、仓位、止损和异常场景。
3. 参数说明：默认值、合法范围、含义和参数间约束。
4. 可视化示例：价格、指标、信号与持仓区间。
5. 验证快照：固定数据区间下的指标和基准对比。
6. 稳健性：参数邻域、分阶段表现、成本敏感性和样本外结果。
7. 局限与失效场景。
8. 版本记录。
9. “在工作台中验证”主操作。

### 7.4 首期策略内容

#### 双均线交叉

- 定位：基础趋势跟随策略。
- 重点解释：滞后性、震荡市反复交易、快慢线约束。
- 默认参数：快线 5、慢线 20、目标仓位 95%。
- 首期稳健性视图：快慢线二维参数热力图、手续费敏感性。

#### 海龟突破

- 定位：通道突破与波动止损策略。
- 重点解释：少数大趋势覆盖多次小亏损、ATR 风控、退出窗口小于入场窗口。
- 默认参数：入场 20、退出 10、ATR 20、止损倍数 2、目标仓位 95%。
- 首期稳健性视图：入场/退出二维参数热力图、ATR 止损敏感性。

## 8. 通用策略工作台

工作台统一分为五个区域：

1. 配置栏
   - 标的、区间、频率、复权、基准、数据源和回测引擎。
   - 初始资金、手续费、滑点、最低佣金、交易价格和整手规则。

2. 策略参数栏
   - 根据策略 Schema 自动生成控件。
   - 参数错误在运行前给出，支持恢复默认值与加载预设。

3. 图表区
   - K 线、指标、信号点、持仓区间、净值与回撤。
   - 图例和提示框显示实际成交而不是仅显示理论信号。

4. 结果区
   - 概览、收益风险、交易质量、交易流水、参数优化、分阶段表现。
   - 总收益、年化收益、最大回撤、波动率、Sharpe、Sortino、Calmar、胜率、盈亏比、Profit Factor、换手和成本。

5. 实验操作区
   - 保存实验、复制配置、导出 JSON/CSV、与其他实验比较。
   - 保存前固定策略版本、数据指纹、引擎、参数和成本假设。

运行状态必须覆盖：未连接、加载数据、排队、运行、完成、失败、取消和回退。任何从专业引擎回退到轻量引擎的情况都必须显著提示，不能静默发生。

## 9. 视觉与交互规范

### 9.1 视觉方向

定位为“研究期刊 × 现代金融工具”，而非传统券商终端。

- 浅色主题：温暖的近白底色、深墨色文字、低饱和蓝作为主色。
- 深色主题：深海军蓝底色，避免纯黑；图表色需重新校准对比度。
- A 股涨跌色可在设置中切换；默认红涨绿跌，并同时使用箭头/符号，避免只靠颜色表达。
- 大面积留白、清晰层级、少量细边框；阴影只用于浮层和关键卡片。
- 数字使用等宽数字特性，正文优先系统中文字体，英文可使用 Inter 类无衬线字体。
- 圆角保持中小尺寸，避免过度胶囊化。

### 9.2 建议设计令牌

| 类型 | 浅色值 | 深色值 |
| --- | --- | --- |
| 页面背景 | `#F5F7FA` | `#0B1220` |
| 卡片背景 | `#FFFFFF` | `#111B2E` |
| 主文字 | `#152033` | `#E8EEF8` |
| 次文字 | `#667085` | `#94A3B8` |
| 品牌色 | `#315EFB` | `#6C8CFF` |
| 上涨 | `#D84A3A` | `#FF7667` |
| 下跌 | `#16805C` | `#43C596` |
| 警示 | `#C97916` | `#F2AE49` |

### 9.3 交互原则

- 桌面优先设计研究工作台，首页和策略详情必须完整适配手机。
- 工作台在窄屏下按“配置 → 图表 → 结果”线性排列，参数栏使用抽屉。
- 任何长任务都显示阶段和可取消状态。
- 空状态说明下一步行动；错误信息提供原因与可执行建议。
- 表格支持键盘访问、固定表头和横向滚动。
- 动效控制在 150–240ms，并尊重系统的减少动态效果设置。
- 正文与背景至少满足 WCAG AA 对比度，交互控件触达尺寸不小于 44px。

## 10. 推荐技术架构

### 10.1 总体结构

建议将项目调整为前后端分层的单仓库：

```text
ai-quant-lab/
├── apps/
│   └── web/                    # React + TypeScript 网站
├── services/
│   └── api/                    # FastAPI 本地研究服务
├── content/
│   ├── strategies/             # 策略说明、风险与版本内容
│   └── snapshots/              # 可公开的只读验证快照
├── contracts/
│   └── schemas/                # 前后端共享 JSON Schema
├── tests/
├── docs/
└── SPEC.md

项目根目录的 quant_lab/         # 首期保留，后续可迁入 Python package
```

### 10.2 前端

建议使用 React、TypeScript、Vite 和声明式路由。理由：

- 保持静态部署能力，与当前纯前端入口迁移路径短。
- 组件化后可复用策略页面骨架和研究控件。
- TypeScript 可约束策略 Schema、回测请求和结果结构。
- 公开首页可通过构建期快照生成，不依赖本地 Python 服务。

前端模块建议：

- `features/strategy-catalog`
- `features/strategy-detail`
- `features/backtest-lab`
- `features/experiments`
- `features/indicator-studio`
- `features/local-service`
- `components/charts`
- `components/ui`
- `lib/contracts`

图表组件应封装为独立适配层，不让策略组件直接操作 Canvas。首期可保留现有绘制逻辑完成迁移，后续再替换为成熟金融图表库。

### 10.3 后端

将当前 `ThreadingHTTPServer` 迁移为 FastAPI，但不改写 `quant_lab` 的核心计算职责：

- API 层只负责参数校验、任务调度、序列化和错误映射。
- `quant_lab` 负责策略信号、回测、指标与优化。
- RQData、RQAlpha 等只在本地服务中初始化，不向浏览器暴露凭据。
- 长时间优化任务使用任务接口，避免阻塞请求。
- 首期默认监听本机地址，并采用严格 CORS 白名单。

### 10.4 双运行模式

#### 公开展示模式

- 网站静态资源 + 已发布的只读策略内容和验证快照。
- 不要求本地服务在线。
- 不展示需要授权的实时行情。
- “运行回测”引导用户连接本地研究服务。

#### 本地研究模式

- 浏览器连接本机 Python API。
- 可使用 RQData、RQAlpha 和本地数据文件。
- 支持保存完整实验、运行参数优化和导出结果。
- 服务状态在全站统一展示，并明确当前数据与引擎能力。

## 11. 策略插件契约

新增策略的关键目标是“注册一次，多处自动生成”。每个策略至少提供一个清单和一个 Python 实现。

### 11.1 策略清单

```json
{
  "id": "ma_cross",
  "version": "1.0.0",
  "name": "MA 双均线交叉",
  "category": "trend",
  "status": "validated",
  "markets": ["CN_STOCK"],
  "frequencies": ["1d", "1w"],
  "summary": "短均线上穿长均线入场，下穿退出。",
  "riskLevel": "medium",
  "implementation": "quant_lab.strategies:MACrossStrategy",
  "params": [
    {
      "key": "fast_window",
      "label": "快线窗口",
      "type": "integer",
      "default": 5,
      "minimum": 2,
      "maximum": 120,
      "step": 1
    },
    {
      "key": "slow_window",
      "label": "慢线窗口",
      "type": "integer",
      "default": 20,
      "minimum": 3,
      "maximum": 260,
      "step": 1
    }
  ],
  "constraints": ["fast_window < slow_window"],
  "capabilities": {
    "backtest": true,
    "optimize": true,
    "rqalpha": true,
    "browserFallback": true
  }
}
```

### 11.2 Python 策略接口

所有策略实现应满足：

- 唯一 `id` 和语义化版本号。
- 参数模型与跨字段校验。
- 声明所需行情字段和最小预热长度。
- 输出统一的信号、目标仓位、原因和可选诊断字段。
- 信号生成函数不执行下单，回测器统一负责成交模拟。
- 不读取决策时点之后的数据。
- 可选提供参数搜索空间、图表覆盖层和 RQAlpha 适配器。

统一信号记录至少包含：

```json
{
  "date": "2026-01-05",
  "signal": 1,
  "targetWeight": 0.95,
  "reasonCode": "FAST_MA_CROSS_UP",
  "reason": "短均线上穿长均线"
}
```

## 12. API 契约

API 统一使用 `/api/v1` 前缀。

| 方法 | 路径 | 用途 |
| --- | --- | --- |
| GET | `/health` | 服务、数据源和引擎能力 |
| GET | `/strategies` | 策略目录 |
| GET | `/strategies/{id}` | 策略清单与参数 Schema |
| GET | `/market-data/bars` | 标准化行情数据 |
| POST | `/backtests` | 创建回测任务 |
| GET | `/backtests/{jobId}` | 查询任务和结果 |
| POST | `/optimizations` | 创建参数优化任务 |
| GET | `/optimizations/{jobId}` | 查询优化状态和结果 |
| POST | `/experiments` | 保存实验 |
| GET | `/experiments` | 实验列表 |
| GET | `/experiments/{id}` | 实验详情 |
| POST | `/experiments/compare` | 对比实验 |

### 12.1 回测请求最小结构

```json
{
  "strategy": {"id": "ma_cross", "version": "1.0.0"},
  "params": {"fast_window": 5, "slow_window": 20, "target_weight": 0.95},
  "universe": ["000001.XSHE"],
  "period": {"start": "2021-01-01", "end": "2025-12-31", "frequency": "1d"},
  "data": {"source": "rqdata", "adjustType": "pre"},
  "execution": {
    "engine": "quant_lab",
    "initialCash": 100000,
    "tradePrice": "next_open",
    "commissionRate": 0.0008,
    "slippageRate": 0.0002,
    "minCommission": 5,
    "lotSize": 100
  },
  "benchmark": "000300.XSHG"
}
```

### 12.2 结果要求

结果必须包含：

- 请求配置快照和配置哈希。
- 策略版本、引擎版本和数据指纹。
- 指标、净值、回撤、交易、信号和运行备注。
- 是否发生引擎回退。
- 警告列表，如样本过短、交易过少、数据缺口或基准缺失。

错误采用统一结构：

```json
{
  "error": {
    "code": "INVALID_STRATEGY_PARAMS",
    "message": "快线窗口必须小于慢线窗口",
    "field": "fast_window",
    "retryable": false,
    "details": {}
  }
}
```

## 13. 数据与实验模型

### 13.1 核心实体

- `StrategyManifest`：策略身份、版本、参数和能力。
- `StrategyContent`：假设、规则、风险、适用场景和变更记录。
- `BacktestRun`：一次执行及其状态。
- `Experiment`：可命名、可复现的回测快照。
- `PublishedSnapshot`：可用于公开页面的只读验证结果。
- `MarketDataFingerprint`：数据源、标的、区间、频率、复权和内容摘要。

### 13.2 存储演进

首期使用本地 SQLite 保存实验元数据和压缩后的结果文件；浏览器 localStorage 仅保存主题、服务地址和未提交表单。

迁移要求：

- 读取现有 `aiQuantLab.indicatorStudio.projects` 和 presets。
- 提供一次性导入预览，用户确认后写入新实验库。
- 保留原 localStorage 数据，迁移成功后也不自动删除。
- 旧配置缺少策略版本时标记为 `legacy`。

## 14. 研究可信度规范

每个公开验证快照必须同时展示：

- 标的范围、时间区间、数据源、频率和复权方式。
- 基准、初始资金、手续费、滑点、最低佣金和交易价格。
- 策略版本、参数、引擎和运行日期。
- 收益、风险、交易质量和样本量。
- “历史回测不代表未来表现”的声明。

首期稳健性检查至少包含：

- 参数邻域而非单点最优。
- 分年度或滚动区间表现。
- 手续费和滑点上调后的敏感性。
- 交易次数过少提示。
- 样本内与样本外区间明确标注。

任何综合评分都只能用于同一策略的实验筛选，并展示构成项，不能作为跨策略的绝对排名。

## 15. 非功能要求

### 15.1 性能

- 首页首屏在常见宽带下快速可用，非关键图表延迟加载。
- 策略目录和内容页面不因本地服务离线而阻塞。
- 超过 1 秒的回测显示进度状态；优化任务可取消。
- 大型净值和行情序列在传输前降采样，但导出保留完整数据。

### 15.2 安全与隐私

- RQData license、token 和本地数据路径不得进入前端、日志、快照或 Git。
- 本地 API 默认只监听 `127.0.0.1`。
- CORS 只允许配置的站点源，不再使用通配符。
- 导入 JSON/CSV 必须做格式、体积和字段校验。
- 策略说明和 AI 解释输出按不可信文本处理，禁止直接注入 HTML。

### 15.3 可测试性

- 策略规则、参数校验、回测成交和指标计算使用 Python 单元测试。
- API 使用契约测试确保字段稳定。
- 前端对策略 Schema 表单、错误态和结果渲染做组件测试。
- 双均线和海龟策略各保留至少一组固定行情的黄金结果。
- 端到端测试覆盖：进入策略 → 修改参数 → 回测 → 保存实验 → 对比。

## 16. 分阶段实施计划

### Phase 0：基线冻结

- 为现有页面和两种策略建立功能清单与黄金回测结果。
- 固定旧配置导出格式，避免迁移过程中无法复现。
- 输出视觉方向稿和首页线框。

完成标准：现有双均线与海龟结果有自动化基准；迁移前能力有清晰清单。

### Phase 1：平台外壳与首页

- 建立新前端工程、路由、设计令牌和响应式页面骨架。
- 完成首页、策略库、方法页和关于页。
- 通过静态策略清单生成双均线和海龟详情页。

完成标准：本地服务离线时，公开内容仍可完整浏览。

### Phase 2：策略注册表与 API

- 建立策略 Manifest、参数模型和注册机制。
- 将后端迁移到版本化 API。
- 让双均线与海龟都通过同一注册表运行。

完成标准：新增策略不需要修改通用页面和通用回测路由。

### Phase 3：通用研究工作台

- Schema 自动生成参数表单。
- 接入行情、回测、结果图表、流水和优化。
- 显式展示引擎、数据源、警告和回退。

完成标准：两个现有策略都能在同一工作台完成全流程验证。

### Phase 4：实验与 Indicator Studio 迁移

- 使用 SQLite 保存实验和对比记录。
- 导入旧 localStorage 项目与 presets。
- 将 Indicator Studio 拆为独立工具模块并接入统一服务状态。

完成标准：旧数据可导入，主要功能无回归，实验可复现。

### Phase 5：质量与发布

- 完成性能、无障碍、安全和跨屏验证。
- 生成并审核公开验证快照。
- 补充部署、备份、升级和新增策略文档。

完成标准：满足本 Spec 的验收标准后再替换旧入口。

## 17. 新增策略标准流程

未来增加一个策略时，只需完成：

1. 创建策略 Manifest 和研究说明。
2. 实现统一 Python 策略接口与参数校验。
3. 按需实现优化空间、图表覆盖层和 RQAlpha 适配器。
4. 添加固定行情单元测试与黄金结果。
5. 生成验证快照并完成风险审阅。
6. 在注册表启用策略；策略库、详情页和工作台自动出现。

禁止通过修改通用页面中的 `if strategy === ...` 来接入新策略；特殊展示需求应通过能力声明或受控扩展点实现。

## 18. 验收标准

平台 2.0 首期完成时，应满足：

- 首页、策略库、策略详情、策略工作台、实验、方法和指标工具均有独立路由。
- 首页在不连接本地 Python 服务时正常工作，不显示虚假实时数据。
- 双均线与海龟策略由同一注册机制发现并渲染。
- 策略表单由 Schema 生成，参数约束在前后端一致执行。
- 回测结果明确显示数据源、引擎、版本、成本、区间和基准。
- 引擎回退可见，失败不会伪装成成功。
- 实验可保存、重新打开、导出和两两比较。
- 旧项目配置可预览导入且不会被自动删除。
- 桌面端完整可用；手机端可浏览首页和详情，并能完成基础回测配置。
- 所有敏感凭据只存在本机受控环境，不进入浏览器或仓库。
- 双均线与海龟的关键回测结果与迁移前黄金结果一致，或有经记录的合理差异。

## 19. 关键决策记录

本 Spec 已默认采用以下决策，实施前可逐项确认：

1. 先做个人研究平台，不做多用户 SaaS。
2. 保留 Python 与 `quant_lab` 为计算核心，前端不重复实现专业回测。
3. 公开网站使用已审核快照，本地服务负责授权数据和完整研究。
4. 新前端采用 React + TypeScript + Vite，旧静态页面渐进迁移。
5. 实验首期存储使用本地 SQLite，不直接引入云数据库和登录系统。
6. 首期策略为双均线和海龟，暂不扩张策略数量，先验证扩展机制。

## 20. 后续设计输入

进入实现前还需最终确认：

- 网站是只在本机使用，还是需要一个公开可访问的展示地址。
- 品牌名称是否继续使用 `AI Quant Lab`，是否需要个人署名或 Logo。
- 首页验证快照使用哪些标的、基准和固定时间区间。
- 深色主题是否首期必需。
- 是否允许公开策略源代码，还是只公开规则和验证结果。
