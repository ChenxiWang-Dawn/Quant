# A 股系统化量化交易系统 Spec

> 状态：Draft v3.0
> 日期：2026-07-24
> 项目代号：`a-share-quant-system`
> 策略代号：`a_share_regime_multifactor_v1`
> 范围：策略、数据、研究、组合、回测、模拟交易、产品、工程、测试、部署、运维与受控实盘前规范；本阶段不实现
> 默认模式：A 股多头、中低频、横截面选股、市场状态约束、周度调仓

## 1. 文档目的

本文定义一套可以被程序唯一解释、完整回测、样本外验证、工程实现、持续运行和审计的 A 股量化交易系统。

本文的目标不是承诺收益，也不是把某种技术形态包装成“必胜战法”，而是规定：

- 策略要赚取什么类型的收益。
- 使用哪些当时可获得的数据。
- 如何建立股票池、信号、组合和订单。
- 如何处理 A 股的停牌、涨跌停、T+1、整手和交易费用。
- 如何证明策略不是未来函数、幸存者偏差或参数过拟合的产物。
- 什么时候允许进入模拟盘、小资金实盘，什么时候必须暂停。
- 各工程模块如何拆分、存储、调用、测试、部署和监控。
- 每个实施阶段的输入、任务、交付物、测试和退出 Gate。

本文是研究与工程规范，不构成个性化投资建议。

### 1.1 独立性声明

`a-share-quant-system` 是独立绿地项目：

- 拥有自己的 Python 包、CLI、API、配置、数据库、任务队列、产物目录和可选 Operator Console。
- 核心系统不导入工作区内其他项目的源代码。
- 不共享其他项目的运行数据库、缓存、策略注册表或实验记录。
- 不要求其他 Web 平台或本地服务在线。
- 外部数据、优化器、统计库和券商能力只能通过本系统定义的适配器进入。
- 项目可以单独安装、测试、部署、备份、恢复和归档。

## 2. 策略定位

### 2.1 一句话定义

在流动性和基本面质量合格的 A 股股票池中，综合质量、估值、中期动量、盈利改善和量价突破信号，构建行业与风格受控的多头组合，并依据市场趋势、宽度和波动状态动态调整总仓位。

### 2.2 核心收益来源假设

策略依赖以下相互独立但可组合的假设：

1. 质量与现金流较好的公司长期发生永久性损失的概率相对较低。
2. 相对估值较低且基本面未恶化的公司存在风险补偿和估值修复机会。
3. 盈利预期、产业趋势和资金再定价不会在一个交易日内完全完成，中期相对强势可能延续。
4. 业绩改善、盈利预测上调和正式披露事件提供比纯价格信号更及时的基本面变化信息。
5. 在行业和市场共振条件下，经过波动收缩后的价格突破可以作为趋势启动或延续的辅助确认。
6. 上述信号在不同市场阶段表现不同，市场状态控制可以降低弱势和高波动阶段的风险暴露。

### 2.3 明确不采用的假设

首版策略不得依赖以下不可验证叙事：

- “主力吸筹”“洗盘”“高度控盘”等无法从公开数据唯一识别的行为解释。
- W 底、圆弧底、上升三角形等依赖人工目测的形态名称。
- “突破后三日成功率超过 90%”等缺少样本、成本和成功定义的经验结论。
- 成交量放大必然代表机构买入。
- 所有股票统一使用 5% 固定止损。
- 单一模型或单一因子可以适应所有市场环境。

## 3. 目标与非目标

### 3.1 研究目标

- 获取相对匹配基准的中长期成本后超额收益。
- 收益来源在质量、估值、动量、事件和突破子信号之间可解释。
- 最大回撤、行业集中、风格漂移和流动性风险处于预设范围。
- 在参数邻域、不同年份和交易成本压力下保持基本稳定。
- 支持从规则基线逐步升级到机器学习排序，但不依赖机器学习才能成立。

### 3.2 运行目标

- 信号频率：日频计算。
- 组合调仓：默认每周一次。
- 预测与持有周期：默认 10 至 20 个交易日。
- 持仓数量：默认 40 只，允许范围 30 至 60 只。
- 方向：只做多。
- 杠杆：首版禁止。
- 卖空：首版禁止。
- 目标股票净敞口：由市场状态决定，默认范围 40% 至 95%。

### 3.3 非目标

- 不追求日内交易和高频收益。
- 不做打板、连板、隔夜情绪接力。
- 不预测单只股票下一交易日涨跌。
- 不使用无法合法、稳定获得的盘口或“资金流向”数据作为核心信号。
- 不在首版接入券商自动下单。
- 不以回测最高收益或最高胜率作为优化目标。

## 4. 冻结决策

以下决策在 v1.0 研究开始前冻结；如需修改，必须新建策略版本并记录原因：

| 项目 | v1.0 决策 |
| --- | --- |
| 市场 | 沪深 A 股普通股票 |
| 方向 | 多头 |
| 频率 | 日频信号、周度调仓 |
| 主框架 | 市场状态约束下的横截面多因子 |
| 基本组合 | Top-K 与受约束优化器并行比较 |
| 主基准 | 中证全指；同时报告中证 500、中证 1000 和股票池等权 |
| 默认持仓 | 40 只 |
| 信号时点 | 交易日收盘后 |
| 最早成交 | 下一交易日 |
| 调参原则 | 训练集和验证集调参，最终测试集只使用一次 |
| AI 地位 | Challenger；规则基线是 Champion 起点 |
| 自动实盘 | 禁止 |

## 5. 系统架构

```text
原始数据
  ↓
时间点正确的数据快照
  ↓
可投资股票池
  ↓
特征与子信号
  ↓
横截面标准化与中性化
  ↓
综合 Alpha 排名
  ↓
市场状态与总仓位
  ↓
组合构建与风险约束
  ↓
订单计划与成交仿真
  ↓
持仓、归因、监控与降级
```

模块之间必须通过版本化数据对象连接，不允许在回测函数内部临时读取未经记录的数据。

## 6. 数据规范

### 6.1 必需数据

| 数据类别 | 最低字段 |
| --- | --- |
| 日行情 | 前收盘、开高低收、成交量、成交额、复权因子 |
| 证券状态 | 上市、退市、停牌、复牌、风险警示、证券类别 |
| 交易规则 | 涨跌停价、最小价位、整手规则、板块规则 |
| 公司行动 | 分红、送转、配股、拆并股 |
| 财务数据 | 报告期、公告日、营业收入、利润、现金流、资产负债表 |
| 行业分类 | 分类版本、生效日期、失效日期 |
| 指数数据 | 历史成分、权重、行情 |
| 交易日历 | 交易日、特殊交易安排 |

### 6.2 可选增强数据

- 业绩预告、业绩快报与正式公告。
- 分析师盈利预测及其历史快照。
- 回购、增持、减持和限售解禁公告。
- 融资融券和股权质押风险。
- 公告文本和新闻文本。
- 逐笔、分钟或盘口数据，仅用于执行模型，不改变日频主策略定义。

可选数据缺失时，策略必须退化到公开可复现的基础版本，而不是静默使用未来数据或当前快照。

### 6.3 时间语义

每个字段至少记录：

- `event_time`：经济事件实际对应的时间。
- `available_time`：研究者最早可以获得该数据的时间。
- `ingested_at`：系统入库时间。
- `revision_id`：修订版本。

特征只能使用决策时点之前已经到达 `available_time` 的数据。

财务数据不得按报告期末日期提前进入特征。公告在收盘后发布时，最早只能影响下一交易日的决策或成交。

### 6.4 数据质量 Gate

进入策略计算前必须通过：

- 证券代码和交易所映射完整。
- 行情日期与交易日历一致。
- 复权因子无异常跳变。
- 停牌日没有伪造成交。
- 历史风险警示和退市状态可重建。
- 历史行业和指数成分未被当前成分回填。
- 财务公告日和修订记录完整。
- 缺失值、极端值和重复数据有审计记录。

任何核心数据 Gate 失败时，当日策略状态为 `data_blocked`，禁止生成正式目标仓位。

## 7. 可投资股票池

### 7.1 基础范围

纳入：

- 沪深交易所上市的人民币普通股票。
- 账户权限允许交易的主板、创业板和科创板股票。

排除：

- B 股、基金、ETF、REIT、债券、可转债和存托凭证。
- 风险警示、退市整理和已公告终止上市股票。
- 上市不足 250 个交易日的股票。
- 决策日停牌或预计执行日不可正常交易的股票。
- 最近 20 个交易日日均成交额不足 5,000 万元的股票。
- 最近 20 个交易日有效交易日不足 15 日的股票。
- 最近收盘价低于 3 元的股票。
- 数据质量或公司行动状态无法确认的股票。

金额与价格阈值必须配置化，并在每次实验中冻结。

### 7.2 流动性与容量

单只股票计划交易金额不得超过过去 20 个交易日日均成交额的 5%。

若组合规模扩大，系统必须先降低目标权重或延长成交周期，不得假设无限流动性。

### 7.3 涨跌停与一字板

- 涨停且缺少可成交卖盘时，不得假设买入成功。
- 跌停且缺少可成交买盘时，不得假设卖出成功。
- 连续一字板期间保持未成交订单状态，并记录风险敞口。
- 因涨跌停未成交的订单不得使用涨跌停价伪造成交。

### 7.4 股票池快照

每个交易日保存：

- 原始证券数量。
- 各排除规则剔除数量。
- 最终可投资数量。
- 剔除原因明细。
- 股票池版本与数据指纹。

## 8. 子信号定义

### 8.1 质量信号

目标：偏好盈利质量和财务稳健性较高的公司，减少永久性损失和财务异常风险。

候选特征：

- ROE TTM 及其稳定性。
- 毛利率和营业利润率稳定性。
- 经营现金流与净利润之比。
- 应计利润。
- 资产负债率和利息保障能力。
- 最近三年亏损次数。
- 非标准审计意见和重大财务异常标记。

质量得分默认权重：25%。

### 8.2 估值信号

目标：在基本面质量可接受的前提下，获取估值风险补偿。

候选特征：

- 盈利收益率。
- 账面市值比。
- 销售收入市值比。
- 自由现金流收益率。
- 企业价值相对经营利润。
- 股息率，仅作为辅助项。

估值指标必须在行业内标准化，并对亏损、负分母和极端值设置明确处理规则。

估值得分默认权重：20%。

### 8.3 中期动量信号

目标：捕捉基本面与资金重新定价的持续性。

候选特征：

- 过去 12 个月跳过最近 1 个月的收益。
- 过去 6 个月跳过最近 1 个月的收益。
- 过去 60 日相对行业收益。
- 过去 20 日相对基准收益。
- 行业和市场回归后的残差动量。

涨停日收益、停牌复牌跳空和极端单日收益必须单独标记，并进行“保留、缩尾、剔除”消融比较，不能默认全部视为可持续动量。

动量得分默认权重：25%。

### 8.4 盈利改善与事件信号

目标：识别基本面增量信息，而不是仅依赖静态财务水平。

基础版特征：

- 营业收入和利润同比改善。
- 经营现金流趋势改善。
- 业绩预告相对上一报告期的变化。
- 正式披露后的盈利超预期代理。

增强版特征：

- 一致预期上调比例。
- 未来年度盈利预测修正。
- 回购、增持和重大订单的结构化事件。

事件数据必须保留原文来源、公告时间和抽取置信度。未经人工或规则验证的大模型抽取结果不得直接进入正式策略。

盈利改善与事件得分默认权重：15%。

### 8.5 量价突破信号

目标：把突破交易转化为客观、可消融的趋势确认特征，不使用主观图形名称。

默认候选条件：

- 收盘价高于过去 60 个交易日最高收盘价。
- 突破幅度超过最小缓冲值。
- 突破前 20 日真实波幅或收益波动处于自身历史较低分位。
- 当日成交额相对过去 20 日成交额中位数适度放大。
- 过去 20 至 60 日相对行业和基准强度为正。
- 非一字涨停，且下一交易日存在现实成交可能。

三日站稳不作为铁律，而作为独立可测试特征：

- `confirmation_0d`：突破收盘后，下一交易日执行。
- `confirmation_2d`：连续两日未跌回原区间。
- `confirmation_3d`：连续三日未跌回原区间。
- `retest_entry`：回踩突破区间后重新走强。

不得预设任何确认方式的胜率。最终选择必须基于样本外成本后期望收益。

量价突破得分默认权重：15%。

### 8.6 拥挤与风险惩罚

以下特征不提供正向 Alpha，只用于降权：

- 最近一个月涨停次数过多。
- 单日成交额和换手率极端放大。
- 过去 20 日波动率和跳空风险过高。
- 流动性快速恶化。
- 股价严重偏离中期均线。
- 应收、存货、质押、解禁或融资风险显著上升。

## 9. 特征处理

### 9.1 横截面处理顺序

1. 根据字段经济含义处理无效值。
2. 使用当日可投资股票池进行缩尾。
3. 缺失值按行业中位数或显式缺失标记处理。
4. 横截面标准化。
5. 对行业和市值暴露进行中性化。
6. 生成子信号得分。
7. 组合子信号并再次检查暴露。

### 9.2 默认缩尾

默认使用横截面 1% 与 99% 分位缩尾。样本过小时使用中位数绝对偏差方法。

### 9.3 缺失值

- 缺失本身具有经济意义时，增加缺失指示变量。
- 缺失比例超过 30% 的特征不得进入正式综合得分。
- 不得使用未来期间值向前填充财务数据。
- 不得跨证券填充时间序列行情。

### 9.4 中性化

默认控制：

- 一级行业。
- 对数流通市值。
- 可选控制 Beta 和波动率。

中性化前后必须保存暴露差异，避免中性化把信号完全消除而无人察觉。

## 10. 综合 Alpha

规则基线的综合得分为：

$$
Alpha_{i,t}
=
0.25 Q_{i,t}
+
0.20 V_{i,t}
+
0.25 M_{i,t}
+
0.15 E_{i,t}
+
0.15 B_{i,t}
-
P_{i,t}
$$

其中：

- Q 为质量得分。
- V 为估值得分。
- M 为中期动量得分。
- E 为盈利改善与事件得分。
- B 为量价突破得分。
- P 为拥挤与风险惩罚。

权重是 v1.0 研究起点，不代表最优参数。

允许的权重调整必须满足：

- 在训练集和验证集内完成。
- 每个子信号权重不超过 35%。
- 任何子信号不得因一次测试期表现差而临时删除。
- 权重调整记录原因、时间和实验编号。
- 最终测试集开始后权重冻结。

### 10.1 必做消融

至少比较：

- 质量与估值组合。
- 纯动量。
- 纯量价突破。
- 不含突破的多因子。
- 完整多因子。
- 完整多因子加市场状态。

若完整模型不稳定地优于简单基线，应保留简单版本。

## 11. 市场状态模型

### 11.1 目的

市场状态只决定总仓位和风险预算，不直接决定单只股票排名。

### 11.2 输入

- 宽基指数相对 60 日和 120 日均线的位置。
- 全市场上涨股票比例。
- 创 60 日新高与新低股票比例。
- 全市场成交额趋势。
- 横截面收益分散度。
- 指数实现波动率和最大回撤。

### 11.3 状态

| 状态 | 典型条件 | 目标净敞口 |
| --- | --- | ---: |
| `risk_on` | 趋势向上、市场宽度健康、波动可控 | 90% 至 95% |
| `neutral` | 趋势分化或宽度一般 | 65% 至 80% |
| `risk_off` | 趋势向下、宽度恶化或波动显著上升 | 40% 至 55% |
| `data_blocked` | 关键数据不完整或异常 | 不生成新风险仓位 |

状态阈值必须通过滚动验证确定，并保持简单。首版禁止使用黑箱模型预测牛熊。

### 11.4 状态切换缓冲

- 状态至少连续两个交易日满足条件后升级风险。
- 风险下降允许更快执行。
- 单日大幅波动可以触发临时降仓，但不能自动恢复。

## 12. 组合构建

### 12.1 基线组合

建立两个并行基线：

1. Top-K 等权组合。
2. 受约束优化组合。

Top-K 等权是可解释基准，优化组合只有在样本外稳定改善风险收益后才可成为默认。

### 12.2 优化目标

组合优化目标为最大化预期 Alpha，同时惩罚风险、换手和交易成本：

$$
\max_w
\left(
w^\top \alpha
-
\lambda w^\top \Sigma w
-
\gamma C(w-w_{old})
\right)
$$

### 12.3 默认约束

| 约束 | 默认值 |
| --- | --- |
| 持仓数量 | 30 至 60 |
| 单股权重 | 不超过 3% |
| 前十大持仓 | 合计不超过 25% |
| 单一行业权重 | 不超过 20% |
| 行业相对基准偏离 | 不超过正负 5 个百分点 |
| 周度单边换手 | 目标不超过 25% |
| 单股成交参与率 | 不超过 20 日均成交额的 5% |
| 现金比例 | 根据市场状态确定 |
| 杠杆 | 禁止 |
| 空头 | 禁止 |

### 12.4 风格约束

至少监控：

- 市值。
- Beta。
- 波动率。
- 价值与成长。
- 流动性。
- 盈利能力。

优化器不得通过隐藏的小盘、低流动性或高 Beta 暴露伪装成 Alpha。

### 12.5 优化失败

当约束无可行解时：

1. 保持已有持仓。
2. 仅执行强制风险退出。
3. 按预设顺序放宽非安全性约束。
4. 记录冲突约束。
5. 禁止静默切换为无限制组合。

## 13. 建仓、加仓与退出

### 13.1 建仓

- 周度调仓时按目标权重生成订单。
- 突破子信号不得单独触发重仓。
- 新股票首次目标权重不超过正常目标权重的 50%。
- 下一次调仓仍保持高排名且可成交时，允许补足目标权重。

这将“试探建仓＋确认加仓”转化为可执行规则，而不是主观临盘决定。

### 13.2 加仓

允许加仓的条件：

- 综合排名仍位于目标持仓区间。
- 市场状态没有降级。
- 行业和单股约束仍满足。
- 价格未出现极端跳空。
- 预计交易成本未超过 Alpha 预算。

禁止因为浮亏而无条件摊低成本。

### 13.3 常规退出

满足任一条件时进入减仓候选：

- 综合排名跌出持仓缓冲区。
- 质量或盈利改善信号显著恶化。
- 行业或风格约束要求降低权重。
- 流动性下降导致容量不合格。
- 达到最大风险权重或组合需要降仓。

使用排名缓冲区降低无效换手。例如建仓阈值为前 40 名，退出阈值可以设置为跌出前 70 名。

### 13.4 风险退出

以下条件可在非调仓日触发风险订单：

- 风险警示、立案调查、退市风险或重大财务异常。
- 价格跌破结构与波动率联合阈值。
- 单股波动率、跳空或流动性风险显著超限。
- 组合或市场状态进入紧急风险模式。

风险订单仍受 T+1、停牌和跌停成交约束。

### 13.5 止损原则

首版不采用统一 5% 止损。止损距离取结构距离与波动率距离中更宽者，并通过仓位控制单笔风险。

$$
StopDistancePct_i
=
\max
\left(
StructureDistancePct_i,\,
k \times \frac{ATR_{20,i}}{EntryPrice_i}
\right)
$$

$$
TargetWeight_i
\le
\min
\left(
WeightCap,\,
\frac{SingleNameRiskBudget}{StopDistancePct_i}
\right)
$$

止损价不是成交保证。回测必须模拟跳空、跌停和延迟成交。

## 14. 订单与执行模型

### 14.1 时点

- 交易日收盘后生成信号和目标仓位。
- 不得使用当日收盘价完成基于该收盘价生成的订单。
- 最早在下一交易日提交订单。

### 14.2 基础成交模型

研究阶段至少提供：

1. 次日开盘成交模型。
2. 次日 VWAP 代理成交模型。
3. 带参与率和冲击成本的分批成交模型。

三种模型必须使用同一组策略信号进行比较。

### 14.3 订单限制

- 买入数量按 100 股整手向下取整。
- 卖出允许处理不足一手的剩余股份。
- 资金不足时按优先级缩减订单，不得生成负现金。
- 开盘跳空超过预设阈值时，可取消追价或转为限价计划。
- 未成交订单不得跨日无限保留，必须有失效时间。

### 14.4 交易成本

成本模型参数化记录：

- 佣金。
- 最低佣金。
- 印花税。
- 过户费和其他规费。
- 买卖价差。
- 滑点。
- 市场冲击。

回测至少运行三组情景：

| 情景 | 用途 |
| --- | --- |
| Base | 当前合理成本假设 |
| Adverse | 滑点和冲击提高约一倍 |
| Stress | 极端流动性与成交恶化 |

具体费率不得硬编码为永久常量，应按生效日期版本化。

## 15. 风险管理

### 15.1 风险层级

```text
数据风险
  ↓
单股风险
  ↓
行业与风格风险
  ↓
组合波动与回撤
  ↓
成交与流动性风险
  ↓
系统与操作风险
```

### 15.2 单股风险

- 权重上限。
- 流动性参与率上限。
- 跳空和涨跌停风险标记。
- 财务、治理和退市风险排除。
- 单股连续未成交天数监控。

### 15.3 组合风险

- 行业集中度。
- 风格暴露。
- 组合 Beta。
- 预期波动率。
- 最大回撤。
- 前十大持仓集中度。
- 组合流动性天数。

### 15.4 回撤机制

相对策略历史净值高点：

| 回撤状态 | 默认动作 |
| --- | --- |
| 低于 8% | 正常运行 |
| 8% 至 12% | 总风险预算降低 20%，禁止提高单股上限 |
| 12% 至 18% | 总风险预算降低 40%，进入策略审阅 |
| 超过 18% | 暂停新增风险，仅允许减仓和必要退出 |

回撤阈值是风险措施，不是策略有效性的证明。阈值应在样本外测试前冻结。

### 15.5 紧急停止

以下情形触发 `kill_switch`：

- 行情或证券状态数据大面积异常。
- 目标仓位与当前持仓差异无法解释。
- 订单金额超过资金或容量限制。
- 单日订单数量或申报频率异常。
- 成交回报与订单状态长时间不一致。
- 组合暴露突破硬约束。
- 程序版本、配置版本或数据版本无法追溯。

## 16. 回测规范

### 16.1 最低历史区间

目标覆盖 2012 年至当前可获得的完整历史，并包含：

- 上涨市场。
- 下跌市场。
- 震荡市场。
- 小盘与大盘风格切换。
- 高波动与流动性收缩阶段。

### 16.2 初始时间切分

建议首版冻结为：

| 区间 | 用途 |
| --- | --- |
| 2012-01 至 2018-12 | 研究与初始参数 |
| 2019-01 至 2021-12 | 验证与有限调参 |
| 2022-01 至 2025-12 | 最终样本外测试 |
| 2026-01 起 | 影子运行或模拟盘，不再回填调参 |

如数据覆盖不足，使用滚动 Walk-Forward；最终测试区间仍必须保持未见。

### 16.3 必须模拟

- 历史上市与退市。
- 历史风险警示。
- 停牌和复牌。
- 涨跌停和一字板。
- T+1。
- 复权与公司行动。
- 整手和现金约束。
- 费用、滑点与市场冲击。
- 未成交和部分成交。
- 财务数据公告延迟和修订。
- 历史行业与指数成分。

### 16.4 基准

同时报告：

- 中证全指。
- 中证 500。
- 中证 1000。
- 当期可投资股票池等权组合。
- 简单质量价值基线。
- 纯动量基线。
- 纯突破基线。

不能只选择事后表现最弱的指数作为基准。

### 16.5 指标

收益类：

- 累计收益。
- 年化收益。
- 年化超额收益。
- 月度和年度胜率。

风险类：

- 年化波动率。
- 最大回撤。
- 下行波动率。
- Beta 与跟踪误差。
- VaR 与尾部损失，仅作描述性风险指标。

风险调整类：

- Sharpe。
- Sortino。
- Calmar。
- 信息比率。

交易类：

- 单边换手。
- 成交率。
- 平均滑点。
- 成本占毛 Alpha 比例。
- 涨跌停未成交率。
- 组合清算天数。

信号类：

- Rank IC 和 ICIR。
- 分层组合收益。
- 因子衰减。
- 子信号相关性。
- 行业和风格暴露。
- Alpha、风险和成本归因。

### 16.6 期望收益

任何“高胜率”结论必须同时展示平均盈利、平均亏损和成本：

$$
ExpectedReturn
=
WinRate \times AvgWin
-
\left(1-WinRate\right) \times AvgLoss
-
Cost
$$

## 17. 稳健性与反过拟合

### 17.1 必做测试

- 参数邻域测试。
- 不同调仓日测试。
- 不同股票池和流动性阈值测试。
- 不同持仓数量测试。
- 不同成交价格模型测试。
- 成本 Base、Adverse、Stress 测试。
- 行业中性化开关测试。
- 涨停日动量处理消融。
- 突破即时、两日、三日和回踩确认比较。
- 极端年份剔除测试。
- 子信号逐个移除测试。

### 17.2 多重检验

- 记录尝试过的全部参数和规则，不只保存成功结果。
- 使用 Deflated Sharpe Ratio、PBO 或适当的多重检验方法评估过拟合风险。
- 禁止在同一最终测试集上反复挑选策略。
- 策略失败结果进入负面结果库。

### 17.3 稳定性优先

若两个版本收益接近，默认选择：

- 参数更少。
- 换手更低。
- 解释更清晰。
- 容量更大。
- 对成本更不敏感。
- 跨时期更稳定。

## 18. 研究晋级 Gate

### 18.1 从研究进入正式回测快照

必须满足：

- 数据质量 Gate 全部通过。
- 无未来函数和当前成分回填。
- 所有信号能从原始数据重建。
- 与至少三个简单基准完成比较。
- 交易成本和未成交逻辑已启用。

### 18.2 从回测进入模拟盘

默认门槛：

- 最终样本外成本后年化超额收益为正。
- 样本外信息比率不低于 0.6。
- 样本外最大回撤不高于 25%。
- Adverse 成本下仍保持正超额收益。
- 收益不依赖单一年份、单一行业或少数涨停交易。
- 参数邻域内主要结论不反转。
- 无单一隐藏风格暴露解释大部分收益。

门槛是研究 Gate，不是收益保证。

### 18.3 从模拟盘进入小资金观察

至少完成：

- 连续 3 至 6 个月影子或模拟运行。
- 信号、订单、成交和持仓可以每日对账。
- 实际可成交价格与回测滑点假设大体一致。
- 数据延迟、停牌、涨跌停和失败恢复经过演练。
- 人工审批策略版本和资金上限。
- 与券商确认程序化交易报告和账户要求。

### 18.4 小资金观察限制

- 初始资金不超过计划策略容量的 10%。
- 不使用杠杆。
- 不自动扩容。
- 每次扩容必须完成容量、滑点和回撤复核。
- 实盘结果不得回写并覆盖原始样本外结论。

## 19. 机器学习升级路径

### 19.1 基线优先

规则多因子是首个 Champion。机器学习只能作为 Challenger：

- 线性或弹性网络排序。
- LightGBM、XGBoost 或 CatBoost 排序。
- 概率模型输出未来超额收益分位。

### 19.2 预测目标

推荐预测未来 10 个交易日的行业与市值中性相对收益排名，而不是直接预测买卖标签。

### 19.3 训练约束

- 使用 Purged Walk-Forward 和 Embargo。
- 特征、标签和股票池均为时间点正确版本。
- 每次训练保存随机种子、依赖、代码版本和数据指纹。
- 测试集不参与特征筛选和超参数调整。
- 模型输出必须经过独立组合层。

### 19.4 晋级条件

机器学习版本必须相对规则基线：

- 样本外成本后收益有稳定增量。
- 最大回撤或尾部风险没有明显恶化。
- 换手和容量在约束范围。
- 至少两个 Walk-Forward 阶段取得改善。
- 特征重要性和模型漂移可以解释。

若不满足，保留为研究结果，不替换 Champion。

## 20. 监控与归因

### 20.1 每日监控

- 数据新鲜度与缺失。
- 股票池变化。
- 市场状态。
- 目标仓位和当前仓位差异。
- 风险约束。
- 未成交和异常订单。
- 组合预估波动与流动性。

### 20.2 每周监控

- 调仓和成本归因。
- 子信号贡献。
- 行业和风格暴露。
- 排名稳定性。
- 预测与实际收益的初步偏差。

### 20.3 每月监控

- 成本后收益和基准差异。
- IC、分层收益和因子衰减。
- 模型或规则漂移。
- 容量与实际滑点。
- 回撤状态。
- 失效条件是否触发。

### 20.4 归因

至少拆分：

- 市场 Beta。
- 行业配置。
- 风格暴露。
- 个股选择。
- 市场状态降仓。
- 交易成本。
- 未成交影响。
- 各子信号边际贡献。

## 21. 失效与降级条件

触发以下任一情况时，策略从 `validated` 降级到 `review_required`：

- 连续 6 个月 Rank IC 显著低于历史区间。
- 12 个月滚动成本后超额收益为负且无法由已知市场状态解释。
- 实际滑点持续超过 Adverse 假设。
- 收益主要转化为小盘、行业或高 Beta 暴露。
- 组合容量低于计划资金规模。
- 财务或事件数据源失去时间点正确性。
- 交易制度、费用或程序化交易监管发生重大变化。
- 市场结构变化使关键特征定义失效。

失效后不得通过临时调参直接恢复，必须创建新实验和新版本。

## 24. 策略能力阶段摘要

本节只描述策略研究能力的递进顺序。完整工程实施路线、任务依赖、交付物和阶段 Gate 见第 38 节。

### Phase 0：数据可信度

- 历史证券状态、复权、行业、财务公告日和交易日历。
- 股票池重建。
- 涨跌停、停牌、T+1 和费用测试。

完成标准：任意历史交易日都能重建当日可投资股票池和已知信息。

### Phase 1：规则基线

- 质量、估值和动量。
- Top-K 等权。
- 周度调仓。
- 基础成交和成本模型。

完成标准：形成最简单、可解释的成本后基线。

### Phase 2：增强信号

- 盈利改善与事件。
- 量价突破。
- 涨停处理和三日确认消融。
- 市场状态模型。

完成标准：证明每个新增模块具有样本外增量价值。

### Phase 3：组合与风险

- 受约束优化。
- 风格暴露。
- 容量、冲击成本和回撤机制。
- 归因与监控。

完成标准：Alpha、风险、成本和成交能够分开解释。

### Phase 4：机器学习 Challenger

- 排序模型。
- Purged Walk-Forward。
- 模型卡和晋级 Gate。

完成标准：机器学习相对规则基线存在稳定、成本后的增量价值。

### Phase 5：模拟与小资金观察

- 每日批处理。
- 信号、订单、成交和持仓对账。
- Kill Switch 和故障演练。
- 程序化交易合规确认。

完成标准：连续运行稳定且与回测假设一致，再由人工决定是否扩大范围。

## 25. 验收清单

在实现前，本文档应满足：

- 策略假设能够被反证。
- 每个信号有唯一、可编程定义。
- 不依赖“主力”“洗盘”等不可观测变量。
- 数据可获得时间和修订规则明确。
- 股票池不存在当前成分回填。
- 组合、风险和执行层与预测层分离。
- A 股成交制度能够进入回测。
- 固定 5% 止损已替换为结构、波动和仓位联合控制。
- 三日站稳被定义为待比较参数，而不是结论。
- 基准、时间切分、成本情景和晋级 Gate 已预先规定。
- 失败结果、修改记录和策略版本可追溯。
- 首版不包含自动实盘交易。

## 28. 系统边界与绿地假设

### 28.1 系统边界

本系统负责：

- A 股研究数据的时间点正确快照。
- 股票池、特征、Alpha、市场状态和目标组合生成。
- 组合级历史回测与真实交易制度仿真。
- 实验、评测、研究 Gate 和负面结果管理。
- 影子运行、模拟盘、订单计划、持仓对账和监控。
- 研究报告、运行报告和经人工确认的订单预览。

本系统暂不负责：

- 无人工确认的券商自动下单。
- 融资融券、期权、期货和跨市场对冲。
- 分钟级高频撮合与盘口预测。
- 投资者适当性、税务或个性化资产配置建议。
- 依赖未授权数据的服务。
- 面向公众的策略展示、荐股、社区、跟单和策略售卖。

### 28.2 绿地项目假设

本项目是独立新系统，实施时按以下前提设计：

- 不导入、不链接、不调用现有 `ai-quant-lab`、`quant_lab` 或其他策略平台代码。
- 不继承已有数据库、API、前端类型、策略注册表、配置格式和回测结果。
- 不承担已有系统的兼容、迁移和回归责任。
- 所有数据契约、计算引擎、会计账本、API、CLI、任务调度和监控均由本项目独立定义。
- 可以使用通用开源库，但必须通过本系统的适配器和契约访问。
- 外部数据源和未来券商接口均视为可替换基础设施，不能渗透核心策略逻辑。
- 核心研究、组合、风险和回测能力在没有 Web 界面时也必须完整运行。

### 28.3 必须建设的能力

系统从零建设：

- 历史证券主数据与每日可投资股票池。
- 横截面特征快照和批量因子计算。
- 多股票 Alpha 排名和行业、市值中性化。
- 市场状态和组合总仓位。
- Top-K 与受约束组合优化。
- 组合级现金、持仓、T+1 可卖数量和公司行动会计。
- 订单、部分成交、涨跌停、停牌和市场冲击仿真。
- 组合级回测、归因、容量和风险评测。
- 每日增量任务、影子组合、订单计划和对账。

### 28.4 独立系统原则

- Core First：数据、策略、组合、风险、执行和账本是核心；页面只是可替换适配层。
- Contract First：先冻结领域对象和时间语义，再实现数据源与计算。
- Portfolio Native：从第一天以多股票组合、现金和订单账本为核心，不从单标的回测器演化。
- Point-in-Time First：任何财务、成分、状态和事件数据必须能够重建历史可见状态。
- Broker Neutral：券商差异封装在适配器，核心系统只认识标准 OrderPlan、Fill 和 PositionSnapshot。
- Reproducible by Default：数据、代码、配置、环境和结果都可追溯。
- Fail Closed：关键数据或账本异常时阻断，不使用猜测值继续运行。

### 30.1 通用身份字段

- `id`：全局唯一标识。
- `version`：语义化版本或内容版本。
- `created_at`：创建时间。
- `as_of`：数据或决策时点。
- `content_hash`：规范化内容哈希。
- `code_commit`：代码提交标识。
- `data_lineage`：上游实体列表。
- `status`：实体状态。

### 30.2 SecurityMaster

| 字段 | 含义 |
| --- | --- |
| `security_id` | 内部稳定证券标识 |
| `symbol` | 当期交易代码 |
| `exchange` | 上交所或深交所 |
| `security_type` | 普通股等类别 |
| `board` | 主板、创业板、科创板 |
| `list_date` | 上市日期 |
| `delist_date` | 退市日期 |
| `currency` | 人民币 |
| `lot_size` | 买入整手 |
| `valid_from` / `valid_to` | 记录生效区间 |

### 30.3 DailyBar

执行和特征数据必须分开表达：

| 字段 | 含义 |
| --- | --- |
| `security_id` | 证券标识 |
| `trade_date` | 交易日 |
| `open_raw` / `high_raw` / `low_raw` / `close_raw` | 未复权价格，用于成交与涨跌停 |
| `volume` / `amount` | 成交量与成交额 |
| `prev_close_raw` | 前收盘 |
| `adjust_factor` | 复权因子 |
| `limit_up_price` / `limit_down_price` | 当日价格限制 |
| `is_suspended` | 是否停牌 |
| `source` | 数据源 |

特征计算使用由原始价格和复权因子派生的连续序列；成交、现金和涨跌停判断必须使用未复权价格。

### 30.4 FundamentalPointInTime

| 字段 | 含义 |
| --- | --- |
| `security_id` | 证券标识 |
| `report_period` | 报告期 |
| `announcement_time` | 首次公告时间 |
| `available_time` | 策略可使用时间 |
| `revision_time` | 修订时间 |
| `field_name` / `value` | 标准化财务字段 |
| `statement_type` | 原始、重述或更正 |
| `source_document_id` | 原公告 |

同一报告期出现更正时必须形成新版本，不覆盖历史当时可见值。

### 30.5 TradingStatus

至少记录：

- 风险警示状态。
- 停复牌状态。
- 退市整理状态。
- 涨跌停制度版本。
- 账户板块权限要求。
- 当日是否可买、是否可卖及原因。

### 30.7 FeatureSnapshot

| 字段 | 含义 |
| --- | --- |
| `feature_set_version` | 特征集版本 |
| `as_of` | 决策时点 |
| `universe_snapshot_id` | 股票池快照 |
| `raw_artifact` | 原始特征 |
| `processed_artifact` | 缩尾、标准化和中性化后特征 |
| `fit_scope` | 当日横截面或训练区间 |
| `quality_report_id` | 质量报告 |

预处理器的拟合范围必须显式保存，防止在测试区间拟合。

### 30.8 AlphaSnapshot

至少包含：

- 各子信号原始值。
- 各子信号标准化值。
- 风险惩罚。
- 综合 Alpha。
- 横截面排名和分位。
- 缺失与降级标记。
- 贡献分解。

### 30.9 RegimeSnapshot

至少包含：

- 输入指标与数据时点。
- 当前状态。
- 前一状态。
- 状态置信度或边界距离。
- 目标净敞口。
- 切换原因代码。

### 30.10 TargetPortfolio

至少包含：

- 当前持仓快照。
- 目标持仓和目标权重。
- 组合构建方法与约束版本。
- 优化前后权重。
- 预估换手、费用、Beta、行业和风格暴露。
- 约束松弛和失败原因。

### 30.11 OrderPlan、Fill 与 PositionSnapshot

`OrderPlan`：

- 证券、方向、目标数量、订单类型和有效期。
- 产生原因、优先级、最大参与率和限价规则。
- 对应目标组合、信号和审批状态。

`Fill`：

- 成交时间、价格、数量、费用、滑点和来源。
- 未成交或部分成交原因。

`PositionSnapshot`：

- 总持仓、T+1 可卖数量、成本、市场价值和未实现盈亏。
- 现金、冻结资金、应收股息和总权益。
- 与券商或回测账本的对账状态。

### 30.12 BacktestRun 与 EvaluationGate

`BacktestRun` 保存：

- 完整配置与哈希。
- 数据、股票池、特征、策略和引擎版本。
- 订单、成交、持仓、净值、归因和警告产物。
- 运行环境、耗时和资源。

`EvaluationGate` 保存：

- 在何时冻结。
- 指标、阈值和比较基准。
- 通过、失败或人工豁免。
- 豁免理由、审批人和影响。

## 31. 数据分层、存储与血缘

### 31.1 数据分层

| 层 | 内容 | 是否可修改 |
| --- | --- | --- |
| Bronze | 数据源原始响应或原始文件 | 只追加 |
| Silver | 标准化证券、行情、财务和状态 | 新版本替代，不覆盖历史 |
| Gold | 股票池、特征、Alpha、组合和评测快照 | 不可变 |
| Serving | API 摘要、内部缓存和运行视图 | 可重建 |

### 31.2 推荐存储

- Parquet：宽表、长时间序列、特征、预测、订单和成交。
- DuckDB：本地批量查询、横截面 Join 和研究分析。
- SQLite：首期任务、实验、版本、状态和审计元数据。
- JSON：小型不可变配置和 Manifest。
- Markdown 或 HTML：研究报告与模型卡。

大型结果不得作为巨型 JSON 塞入 SQLite。

### 31.3 分区与主键

- 日行情按 `trade_year` 和 `exchange` 分区。
- 财务数据按 `announcement_year` 分区。
- 特征按 `feature_set_version` 和 `as_of_date` 分区。
- 订单、成交和持仓按 `run_id` 和日期分区。
- 每张表声明唯一键和重复处理策略。

### 31.4 快照与哈希

- 原始文件保存来源、请求参数、时间和校验和。
- Silver 与 Gold 产物使用规范化排序后计算内容哈希。
- 相同输入和代码产生相同哈希时允许复用。
- 上游快照改变后，下游缓存自动失效。
- 研究与运行报告必须能追溯到不可变 Gold 快照。

### 31.5 数据源降级

- 数据源切换必须产生新的 `source_profile_id`。
- 不同来源结果不能无提示拼接。
- 核心财务或证券状态缺失时阻断正式运行。
- 可选事件数据缺失时允许降级，但 Alpha 快照必须记录缺失模块。
- 研究报告必须说明数据源和降级状态。

### 31.6 备份

- 元数据数据库每日备份。
- Gold 产物和已发布报告按版本备份。
- Bronze 原始数据按数据许可和重建成本决定保留周期。
- 每季度执行一次恢复演练，并记录恢复时间与缺失项。

## 32. 计算引擎与模块契约

### 32.1 UniverseEngine

输入：

- SecurityMaster。
- TradingStatus。
- DailyBar 流动性统计。
- 账户能力配置。
- 股票池规则版本。

输出：

- UniverseSnapshot。
- 全量排除明细。
- 质量报告。

要求：

- 同一输入幂等。
- 每条规则可单独开关做敏感性测试。
- 规则顺序不应改变最终集合；若影响原因统计，应有固定优先级。

### 32.2 FeatureEngine

每个特征插件声明：

- 唯一名称和版本。
- 所需字段。
- 最小回看窗口。
- 可获得时间要求。
- 缺失处理。
- 横截面或时间序列计算类型。
- 输出单位和方向。
- 单元测试和黄金样例。

特征函数不得直接读取数据库；由执行器提供已裁剪到决策时点的数据视图。

### 32.3 AlphaEngine

职责：

- 调用特征集。
- 执行缩尾、标准化和中性化。
- 计算子信号、惩罚和综合 Alpha。
- 保存贡献分解与警告。

禁止：

- 在 Alpha 层生成订单。
- 根据测试期收益临时改变权重。
- 隐藏缺失因子的自动再归一化。

### 32.4 RegimeEngine

- 只输出状态和净敞口范围。
- 不改变股票横截面排序。
- 状态切换使用版本化阈值和缓冲。
- 数据不足时进入 `data_blocked`，不能默认 `risk_on`。

### 32.5 PortfolioEngine

提供统一接口：

```text
build_target_portfolio(
  alpha_snapshot,
  regime_snapshot,
  current_positions,
  benchmark_exposures,
  constraints,
  cost_model
) -> TargetPortfolio
```

实现：

- `TopKEqualWeightBuilder`。
- `ConstrainedOptimizerBuilder`。

任何新优化器必须在固定 Alpha 快照上与 Top-K 基线比较。

### 32.6 RiskEngine

分为：

- 事前风险：目标组合约束、容量、Beta、行业和波动。
- 事中风险：订单金额、参与率、T+1、涨跌停和现金。
- 事后风险：实际持仓、偏离、回撤、滑点和异常。

硬约束失败必须阻断；软约束失败可以警告或降权，但不能静默忽略。

### 32.7 ExecutionSimulator

至少支持：

- 次日开盘。
- 次日 VWAP 代理。
- 参与率限制下的分批成交。
- 部分成交和未成交。
- 停牌、涨跌停、一字板和跳空。
- 费用、最小佣金、滑点和冲击成本。
- T+1 可卖数量。

执行模型与 Alpha 模型完全分离，同一目标组合可以重复使用不同执行模型评测。

### 32.8 PortfolioBacktestEngine

新引擎采用事件顺序：

```text
盘前公司行动与证券状态
→ 开盘可交易性
→ 执行昨日订单
→ 日内成交与费用
→ 收盘估值
→ 更新数据与特征
→ 生成下一交易日目标组合和订单
→ 保存账本与快照
```

任何一天都必须满足会计恒等式：

$$
Equity_t
=
Cash_t
+
\sum_i Position_{i,t} \times Price_{i,t}
+
Receivables_t
$$

### 32.9 EvaluationEngine

输入固定的 BacktestRun 或 ShadowRun，输出：

- 收益、风险、交易和信号指标。
- 基准、分期、成本和参数敏感性。
- 行业、风格、个股和成本归因。
- Gate 结果。
- 可发布研究报告。

评测不得重新运行或修改策略参数。

## 33. API 与任务契约

- 统一前缀 `/api/v1`。
- 创建长任务返回 `202 Accepted` 和 `jobId`。

### 34.3 调度时间

- 所有时间使用 `Asia/Shanghai`。
- 数据源到达时间通过配置管理，不假设收盘后立即完整。
- 任务默认等待行情、证券状态和当日公告达到完整性门槛。
- 数据延迟超过运行窗口时进入 `data_blocked`，禁止使用昨日数据伪装今日结果。
- 节假日、临时休市和半日特殊安排由交易日历驱动。

### 34.4 幂等与重跑

- 每个任务的幂等键由业务日期、配置版本和上游哈希组成。
- 同一幂等键已成功时直接复用。
- 上游数据修订触发新版本，不覆盖旧结果。
- 失败任务只从最近成功检查点重跑。
- 人工重跑必须记录操作者和原因。

### 34.5 并发与资源

- 同一数据快照只允许一个写任务。
- 研究回测可以并行，但需设置 CPU、内存和总任务数上限。
- 每日正式增量任务优先级高于研究任务。
- 资源不足时暂停低优先级回测，不得跳过数据质量 Gate。

### 34.6 任务产物

每个任务必须生成：

- 不可变请求。
- 上游依赖和哈希。
- 开始、结束、耗时和资源。
- 状态、进度、警告和错误。
- 输出产物索引。
- 日志摘要和可重试判断。

## 36. 测试与验证策略

### 36.1 单元测试

数据：

- 代码标准化。
- 交易日历。
- 复权因子。
- 财务 `available_time`。
- 风险警示和退市状态。

特征：

- 回看窗口。
- 缺失值。
- 缩尾和标准化。
- 行业和市值中性化。
- 涨停日处理。

组合：

- Top-K。
- 权重归一。
- 行业和单股约束。
- 换手缓冲。
- 优化不可行。

执行：

- 整手。
- 最低佣金。
- T+1 可卖。
- 部分成交。
- 涨跌停和停牌。
- 公司行动。

### 36.2 属性测试

必须验证：

- 只多组合不存在负持仓。
- 总目标权重不超过市场状态上限。
- 现金和证券总价值满足会计恒等式。
- 成本提高时，同一成交轨迹净收益不能提高。
- 禁止买入时成交数量为零。
- 无新信号和公司行动时，持仓数量不应自行变化。
- 相同输入、配置和代码产生相同哈希和结果。

### 36.3 黄金数据集

建立一个小型、可人工计算的数据集，至少包含：

- 8 至 12 只股票。
- 约 80 个交易日。
- 正常交易、停牌、涨停、跌停和一字板。
- 分红、送转和代码变更中的至少两种。
- 风险警示与股票池剔除。
- 财务公告和更正。
- 行业分类变化。

黄金结果覆盖：

- 股票池。
- 特征。
- Alpha。
- 目标组合。
- 订单和成交。
- 现金、持仓和净值。
- 费用和归因。

### 36.4 未来数据泄漏测试

必须自动化：

- 修改未来行情不改变过去特征。
- 推迟公告 `available_time` 后，过去特征不再使用该数据。
- 使用当前行业或当前指数成分回填历史时测试失败。
- 预处理器不能在验证和测试区间拟合。
- 测试集指标不能被参数搜索器读取。
- 当日收盘信号不能按当日收盘成交。

### 36.5 A 股制度场景测试

最低场景：

1. 信号日涨停，次日仍一字涨停，买单持续无法成交。
2. 持仓次日跌停，卖单无法成交且风险继续计价。
3. 当日新买股票触发风险信号，但 T+1 不可卖。
4. 股票停牌数日后跳空复牌。
5. 分红除权后权益保持会计一致。
6. 送转后持仓数量和价格同步调整。
7. 风险警示制度变化按历史规则生效。
8. 退市或长期停牌资产按预设处置规则处理。

### 36.6 契约测试

- Pydantic 与 JSON Schema 字段一致。
- Python、CLI、API 与可选 TypeScript 类型由同一 Schema 生成或校验。
- API 错误代码稳定。
- 大型产物分页和下载完整。
- CLI、API 与 Operator Console 契约保持一致。

### 36.7 集成测试

- 数据快照到股票池。
- 股票池到特征和 Alpha。
- Alpha 到组合和风险。
- 组合到订单、成交和账本。
- 回测到评测、报告和发布。
- 每日任务失败后的断点恢复。

### 36.8 端到端测试

覆盖：

1. 选择数据快照。
2. 冻结研究配置。
3. 创建回测。
4. 查看进度和警告。
5. 查看成本后结果和归因。
6. 运行稳健性套件。
7. 评估 Gate。
8. 保存实验和负面结果。
9. 晋级到影子运行。
10. 生成模拟订单并完成持仓对账。

### 36.9 回归测试

- 规则 Champion 和黄金账本结果保持。
- 组合引擎每次修改运行固定组合黄金测试。
- 费用、复权、公司行动和指标算法变更必须产生版本说明。
- 性能回归超过预算时阻断合并。

## 37. 非功能、安全与运维规范

### 37.1 性能预算

在参考环境 8 核 CPU、32GB 内存、SSD、特征缓存已建立的条件下：

| 操作 | 目标 |
| --- | --- |
| API 创建任务 | 500 毫秒内返回 |
| 单日增量股票池与特征 | 10 分钟内 |
| 单日组合与风险 | 60 秒内 |
| 2012 至 2025 基线组合回测 | 60 分钟内 |
| 实验摘要页面 | 2 秒内可交互 |
| 取消长任务 | 10 秒内进入取消状态 |

冷启动、全量数据重建和机器学习训练单独记录，不与增量预算混淆。

### 37.2 可用性

- 核心批处理与影子任务不依赖 Operator Console 在线。
- 每日正式任务支持从检查点恢复。
- 元数据数据库损坏时可从备份和产物索引恢复。
- 数据源短暂失败采用有限重试和指数退避。
- 禁止无限重试造成重复请求或费用。

### 37.3 安全

- 数据源、模型和未来券商凭据只在服务端。
- 本地服务默认监听 `127.0.0.1`。
- 云端使用最小权限身份和严格 CORS。
- 文件导入限制类型、大小、路径和内容。
- 下载文件名和路径不得由用户直接控制。
- 日志不得包含密钥、完整账户号或受限数据。
- 报告导出经过字段白名单和敏感信息检查。

### 37.4 权限

首期角色：

| 角色 | 权限 |
| --- | --- |
| Viewer | 查看获准的研究与运行结果 |
| Researcher | 创建数据、特征、回测和实验 |
| Reviewer | 审阅 Gate、发布和影子晋级 |
| Operator | 运行每日任务、处理告警和对账 |

个人项目中可由同一人承担多个角色，但审批动作仍需单独记录。

### 37.5 审计

必须审计：

- 配置创建和修改。
- 测试集查看。
- Gate 冻结、失败和豁免。
- 策略晋级、暂停和回滚。
- 影子订单生成、修改和取消。
- 数据修订和任务重跑。
- 报告导出、归档和撤回。

### 37.6 可观测性

指标：

- 任务成功率、耗时和积压。
- 数据到达延迟和缺失率。
- 股票池数量异常。
- 特征缺失、分布和漂移。
- Alpha IC 和排名稳定性。
- 组合约束、换手和容量。
- 成交率、滑点和未成交。
- 账本对账差异。
- API 延迟和错误率。

日志：

- 结构化 JSON。
- 带 `run_id`、`job_id`、`as_of` 和版本字段。
- 研究日志和运行日志分开。

告警：

- 分为 `info`、`warning`、`critical`。
- `critical` 必须阻断正式影子决策或订单计划。
- 同类告警聚合，避免告警风暴。

### 37.7 运行手册

至少提供：

- 数据延迟。
- 股票池异常收缩。
- 组合无可行解。
- 回测会计不平。
- 影子持仓对账失败。
- 费用或交易规则变更。
- 策略回撤触发。
- 服务升级和回滚。
- 备份恢复。

### 37.8 版本与发布

- 采用语义化版本。
- 策略、数据契约、特征集、执行模型和评测套件独立版本化。
- 破坏性变更必须生成迁移说明。
- 发布前运行完整黄金、泄漏、制度场景和 E2E 测试。
- 每次发布可回滚到上一个已验证版本。

### 37.9 合规边界

- 实施前重新核对交易所、券商和数据许可要求。
- 程序化交易报告由券商确认，不由系统自行判断为“无需报告”。
- 研究和影子运行不得绕过账户权限或市场规则。
- 首版不自动提交订单、不自动扩大资金、不承诺收益。

- 每日人工审批订单。
- 每日对账，差异未解决前禁止新风险。
- 实际滑点超过 Stress 假设时暂停。
- 策略、数据、规则或费用变更必须重新审批。

交付物：

- 实盘前检查单。
- 每日审批与对账。
- 实际成交成本报告。
- 月度策略与风险审阅。
- 扩容或暂停决策记录。

退出 Gate：

- 至少完成一个预设观察周期。
- 成交、成本、风险和操作符合假设。
- 扩容只能由人工决策，不由短期收益触发。

## 41. Definition of Done

### 41.1 单个功能

功能只有同时满足以下条件才算完成：

- 有清晰输入、输出和失败状态。
- 有版本化契约。
- 有单元测试和至少一个边界测试。
- 有日志、指标和错误代码。
- 有文档和示例。
- 不引入未来数据。
- 不绕过风险约束。
- 代码审阅和测试通过。

### 41.2 单个数据集或特征

- 字段和经济含义明确。
- `event_time` 与 `available_time` 明确。
- 来源、许可和版本明确。
- 缺失、修订和极端值规则明确。
- 黄金结果和泄漏测试通过。
- 可以从上游快照重建。

### 41.3 单个策略版本

- Manifest、假设、参数、数据、组合和执行版本固定。
- 样本内、验证、测试和影子区间标记。
- 基准、成本和风险完整。
- 消融和稳健性完成。
- Gate 有结论。
- 失败条件和监控明确。
- 不能因为测试期结果修改后仍沿用原版本。

### 41.4 单个发布版本

- 所有 P0 自动化测试通过。
- 数据库升级和回滚通过。
- 无敏感信息进入仓库、CLI 输出、API 响应或 Operator Console。
- 性能预算满足或有批准豁免。
- 发布说明、已知问题和回滚步骤完整。
- 导出报告完成敏感信息检查和人工审阅。

## 42. 风险登记表

| 风险 | 影响 | 缓解 |
| --- | --- | --- |
| 财务数据存在未来函数 | 回测失真 | PIT 数据、公告时间、泄漏测试 |
| 当前股票名单回填历史 | 幸存者偏差 | SecurityMaster 与每日股票池 |
| 涨跌停按理想价格成交 | 高估收益 | 订单簿约束、未成交和一字板测试 |
| 从单标的模型起步后再硬扩展 | 架构脆弱 | 从第一天建设组合原生引擎 |
| 规则和成本频繁变化 | 结果不可复现 | 生效日期与 Profile 版本 |
| 技术形态过拟合 | 样本外失效 | 客观定义、消融、多重检验 |
| 组合优化器制造隐藏暴露 | 假 Alpha | Top-K 基线、风格约束和归因 |
| 数据源临时缺失 | 错误决策 | data_blocked、双源校验、有限降级 |
| 影子账本不一致 | 运行风险 | 每日对账、Kill Switch |
| 测试集反复查看 | 研究污染 | 测试预算和审计 |
| 策略规模超过容量 | 滑点吞噬收益 | ADV 约束、冲击模型和扩容 Gate |
| AI 模型替代研究纪律 | 黑箱失控 | Champion-Challenger 与人工晋级 |
| 自动化超出合规边界 | 合规风险 | 本地优先、人工审批、券商确认 |
| 文档与实现漂移 | 错误理解 | 契约生成、ADR 和发布检查 |

## 43. 系统最终验收标准

系统进入影子运行前必须满足：

- 能重建 2012 年以来任意样例日的可投资股票池。
- 所有财务特征按当时可获得时间生成。
- 五类子信号、惩罚、综合 Alpha 和市场状态可解释。
- Top-K 和优化组合可以在固定 Alpha 上比较。
- 组合级账本通过 T+1、涨跌停、停牌、公司行动和未成交测试。
- 信号在收盘后产生，最早下一交易日成交。
- 费用、滑点、冲击和容量均进入结果。
- 回测同时展示多个基准、成本后结果和归因。
- 最终测试区间只按预算查看。
- 参数邻域、成本压力、股票池和调仓日稳健性完成。
- Gate 失败时无法晋级。
- 每日任务幂等、可恢复、可监控。
- 关键数据或会计异常触发 Kill Switch。
- Operator Console 和导出报告不泄露私有数据和凭据。
- 规则 Champion 和组合黄金账本没有未解释回归。

系统进入有限资金观察前还必须满足：

- 影子运行达到预设观察期。
- 影子订单、成交和持仓每日对账。
- 实际可成交性与回测假设差异可解释。
- 运行手册、备份恢复和告警演练通过。
- 订单预览、审批和审计完整。
- 券商、账户、程序化交易报告和费用要求已确认。
- 用户另行明确授权资金、账户和风险上限。

## 45. v3.0 关键优化决策

相较之前的策略草案，本版本明确：

1. 项目被定义为完全独立的绿地量化交易系统，不依赖任何已有平台、代码、数据库或 API。
2. 从第一天建设组合原生回测引擎，不从单标的回测器演化。
3. 未复权价格负责成交，复权序列负责特征，避免价格与会计混用。
4. 时间点正确数据、历史股票池和 A 股成交制度列为最先建设的 P0。
5. 突破、三日确认和涨停日处理全部进入消融，不作为预设真理。
6. 规则多因子是 Champion，机器学习只能作为复用同一组合和成交层的 Challenger。
7. 研究、回测、影子、订单预览和有限资金观察采用逐级 Gate。
8. 数据、契约、特征、策略、组合、执行和评测独立版本化。
9. 每日运行使用幂等 DAG、检查点、监控和 Kill Switch。
10. 所有实施阶段均包含入口、任务、产物、测试和退出标准。
